# 玻璃杯AI生成可视化设计系统 — 部署与运行文档

本系统基于 Stable Diffusion WebUI（本地文生图 / 图生图）实现"用户文本驱动的玻璃杯个性化设计"，
并使用 **MOPSO 多目标微粒群算法** 对玻璃热成型工艺参数进行寻优，
同时优化 **壁厚均匀度、成品废品率、加工能耗、耐热安全性** 四项相互冲突的目标。

> **图片生成默认且仅支持本地 SD WebUI（A1111 带 --api 参数）**，
> 不支持在线 Pollinations 模式。

---

## 一、环境要求

| 项目 | 要求 |
|------|------|
| 操作系统 | Windows 10 / 11（脚本按 Windows 编写；Linux/macOS 亦可手动运行） |
| Python | 3.9 及以上（开发环境为 Python 3.14） |
| 网络 | **本地 SD WebUI 模式**：完全离线，无需联网 |
| 显卡 | **SD WebUI 模式需 NVIDIA GPU（≥4GB 显存）** |

## 四、运行方式

### 方式 A：双击启动（推荐）

直接双击项目根目录下的 **`run.bat`**，脚本会自动：

1. 切换到脚本所在目录；
2. 检测 Python 是否安装；
3. 检测并按需安装依赖；
4. 启动后端服务；
5. 在服务就绪后自动用默认浏览器打开 `http://localhost:5000`。

命令行窗口会实时显示日志，**关闭窗口或按 `Ctrl+C` 即可停止服务**。

### 方式 B：命令行启动

```powershell
python app.py
```

启动成功后终端会显示：

```
* Running on http://127.0.0.1:5000
```

然后手动在浏览器访问 [http://localhost:5000](http://localhost:5000)。

> 注意：数据文件路径 `data/designs.json` 是相对"当前工作目录"的，
> 因此请务必在项目根目录下运行 `python app.py`（`run.bat` 已自动处理）。

## 七、图片生成模式说明

系统仅支持**本地 Stable Diffusion WebUI（A1111 带 --api）** 进行图片生成，不支持在线 Pollinations 或其他云端服务。

### 输入合规预检（input_guard）

系统会在调文生图前**先扫一遍用户 prompt**（`src/services/input_guard.py`），命中工业负面词
（变形/破损/裂纹/碎裂/脏/丑/坏 等 29 个中英文词）时，
**不调 SD WebUI**，直接返回 4 张内联 SVG 灰色提示图，不污染数据库。
正常风格描述（简约、北欧、ins风、复古做旧等）不会被误判。

### 启动流程（SD WebUI 模式）

适用于已下载 B 站**秋葉 A绘世整合包**（或其他基于 AUTOMATIC1111 WebUI 的发行版）的用户。
完全离线、生成速度快、不受第三方服务限流。

#### 1. 准备工作

| 项目 | 要求 |
|------|------|
| 整合包路径 | `F:\sd-webui-aki-v4.10\`（按实际修改 `check_env.bat` 顶部的 `SD_LAUNCHER`） |
| 模型文件 | `stable-diffusion-v1-5-pruned-emaonly.safetensors` 放入整合包的 `models/Stable-diffusion\` |
| 启动参数 | `webui-user.bat` 的 `set COMMANDLINE_ARGS=` 行末尾**必须包含** `--api`（整合包默认已带，可在启动器内确认） |
| 端口 | 默认 7860（被占用时需手动改） |

#### 2. 启动流程

SD WebUI 模式**只有一个入口脚本**：`run.bat`。

**整体启动顺序**（重要）：

```
[1] 手动启动 SD WebUI（先于 run.bat）
    ↓
[2] 双击 run.bat（自动检测 WebUI + 启 Flask）
    ↓
[3] 浏览器自动打开 http://localhost:5000
```

**第 1 步：手动启动 SD WebUI**

1. 打开 A绘世启动器：`F:\sd-webui-aki-v4.10\A绘世启动器.exe`
2. 确认「高级选项 → `--api`」已勾选
3. 点「一键启动」
4. 等控制台出现 `Running on local URL: http://127.0.0.1:7860` 后**保持窗口开着**

**第 2 步：双击 `run.bat`**

脚本会按顺序执行：

1. `STEP 1` 检查 Python 是否在 PATH
2. `STEP 2` 检查依赖（flask、numpy、requests），缺失自动装
3. `STEP 3/4` 检查 SD WebUI 状态（7860 端口 + API 探活），输出 `[OK]/[WARN]`
4. `STEP 4/4` 60 秒内轮询等待 WebUI 就绪，每 2 秒显示 `... waiting 2s / 60s`
   - 提前就绪 → `[OK] SD WebUI is now ready!`
   - 60 秒未就绪 → `[TIMEOUT] Flask will start in MOCK mode`
   - Ctrl+C 跳过 → 立即启 Flask
5. 启 Flask，自动打开浏览器到 `http://localhost:5000`

**典型输出（WebUI 已启动）：**

```
[STEP 3/4] Checking SD WebUI status ...
[OK] Port 7860 in use, verifying API endpoint ...
[OK] SD WebUI API ready. Will use REAL image generation.

[STEP 4/4] Waiting for SD WebUI on http://127.0.0.1:7860/sdapi/v1/options
[OK] SD WebUI is now ready!

Starting backend server ...
✓ 已连接 SD WebUI: http://127.0.0.1:7860/sdapi/v1/txt2img
 * Debug mode: off
```

> 与旧版 `start_all.bat` 的区别：现在**必须先手动启动 A绘世启动器**（步骤 1），再跑 `run.bat`（步骤 2）。
> `run.bat` 只负责检测和等待，不会自动拉起 A绘世启动器（避免启动器重复实例冲突）。

控制台会显示：

```
[步骤 1/4] 启动 A绘世启动器: F:\sd-webui-aki-v4.10\A绘世启动器.exe
            请在弹出的窗口里点击"一键启动"按钮
[步骤 2/4] 等待 SD WebUI 就绪 (端口 7860，最长 300s)...
            等待中... 5s
            等待中... 10s
[信息] SD WebUI 已就绪 (耗时 32s)
[步骤 3/4] 校验 API 端点 /sdapi/v1/options ...
[信息] API 端点正常 (HTTP 200)
[步骤 4/4] 启动 Flask 服务...
 * Running on http://127.0.0.1:5000
```

Flask 启动后看到的连接信息是 `✓ 已连接 SD WebUI: http://127.0.0.1:7860/sdapi/v1/txt2img`，
即进入真实推理模式；显示 `⚠ SD WebUI 不可用，使用模拟模式` 则进入 mock（picsum 占位图）。

#### 3. 关闭顺序

为避免显存残留，**按以下顺序关闭窗口**：

1. Flask 命令行窗口 → `Ctrl+C` 或直接关；
2. A绘世启动器窗口；
3. SD WebUI 的 cmd 窗口（看到 `Shutting down` 后再关）。

#### 4. 常见报错速查

| 控制台信息 | 原因 | 解决 |
|------------|------|------|
| `API 端点返回 HTTP 404` | 启动参数没加 `--api` | 编辑 `webui-user.bat`，在 `set COMMANDLINE_ARGS=` 行末尾加 `--api` 后重启 |
| `等待中... 60s 超时` | GPU 驱动问题 / 整合包启动失败 | 手动启动 WebUI 看完整报错 |
| Flask 进入 mock 模式 | 7860 没起来 / API 未开 | 手动启 A绘世启动器并勾 `--api`，再重启 `run.bat` |

#### 5. 环境自检（check_env.bat）

如果 `run.bat` 启动失败（持续 mock 模式或报路径错），**先别盲目重启**，双击项目根目录的 **`check_env.bat`** 做一次独立环境自检，
它会按顺序验证三件事，**5 秒内就能定位是路径、端口还是 API 的问题**：

| 步骤 | 检查项 | 失败原因 | 一行解决 |
|------|--------|----------|----------|
| 1/3 | 整合包启动器 `F:\sd-webui-aki-v4.10\A绘世启动器.exe` 是否存在 | 路径写错 / 整合包被移动 | 编辑 `check_env.bat` 顶部 `SD_LAUNCHER` 为正确路径 |
| 2/3 | 7860 端口占用情况 | 别的程序占了端口（QQ / 迅雷 / VMware 等） | `taskkill /F /PID <占用进程的 PID>`，或在 SD WebUI 启动器内改端口 |
| 3/3 | `GET /sdapi/v1/options` 返回码 | 见下方「返回码含义」 | 见下方 |

**返回码含义：**

| HTTP 码 | 含义 | 处理 |
|---------|------|------|
| `200` | WebUI 已在运行且 API 已开启 | 双击 `run.bat` 即可，脚本会自动跳过等待 |
| `404` | WebUI 页面在跑但没启用 API | 编辑 `webui-user.bat`，在 `set COMMANDLINE_ARGS=` 末尾加 `--api` 后重启 WebUI |
| `000` | WebUI 没启动 | 手动双击 `A绘世启动器.exe` → 点"一键启动"，启动后再跑 `check_env.bat` 复验 |

**典型使用场景：**

- 改了整合包路径后想确认配置对不对 → 跑一次 `check_env.bat`，看步骤 1
- `run.bat` 报"等待 60s 超时"想看是哪个进程占 7860 → 跑 `check_env.bat`，看步骤 2 的占用详情
- Flask 跑起来了但一直进 mock 模式 → 跑 `check_env.bat`，看步骤 3 的 HTTP 码

> 脚本是**只读**的，不会启动任何程序也不会修改文件，可以反复跑。

#### 6. 运行中诊断（diag.bat）

如果 Flask 和 SD WebUI 都已经启动，但**浏览器还是加载慢**或**访问出错**，用 `check_env.bat` 已经晚了
（它只查静态环境）。这时用项目根目录的 **`diag.bat`** 做一次**运行状态诊断**，30 秒内告诉你问题在哪：

| 步骤 | 检查项 | 期望 | 异常意味着 |
|------|--------|------|------------|
| 1/3 | 5000 端口 Flask 进程数 | **1 个** LISTENING | >1 = 多个 Flask 在抢端口（debug mode fork worker）；0 = Flask 没启 |
| 1/3 | 7860 端口 SD WebUI 进程数 | **1 个** LISTENING | 0 = WebUI 没启；>1 = 多份 WebUI |
| 1/3 | python.exe 总数 | **2 个**（Flask + WebUI） | 多了 = 有进程残留；少了 = 缺一个服务 |
| 2/3 | `GET /`（Flask） | HTTP `200` | `0` = Flask 没启；`000` = 连不上 |
| 2/3 | `GET /sdapi/v1/options` | HTTP `200` | `404` = WebUI 没加 `--api`；`000` = WebUI 没起或还在加载 |

**典型输出（健康状态）：**

```
[1/3] Ports and processes
  TCP    0.0.0.0:5000           LISTENING       30060
  TCP    127.0.0.1:7860         LISTENING       42552
  python.exe                   42552   925,304 K
  python.exe                   30060    36,360 K

[2/3] Endpoint probe
  Flask  http://127.0.0.1:5000/             HTTP 200
  SD API http://127.0.0.1:7860/sdapi/v1/... HTTP 200

[3/3] Diagnosis
[OK] Both services healthy. Browser should load fast.
```

**典型输出（5000 端口冲突）：**

```
[1/3] 5000  出现多个 LISTENING
[3/3] [PROBLEM] Flask has multiple processes, port conflict.
     Common fix: kill extra processes:
       taskkill /F /IM python.exe /FI "WINDOWTITLE eq Flask*"
```

**典型使用场景：**

- 浏览器"白屏一直转圈" → 跑 `diag.bat`，看 5000 端口进程数（多半是上次的 Flask 没杀干净）
- 浏览器能打开但生成图片报 500 → 看 SD API 的 HTTP 码（404 = 没加 `--api`）
- 浏览器能开能生图但都很慢 → 看 `python.exe` 内存占用（925 MB 正常，>2 GB = 模型重载了多次）

**与 check_env.bat 的区别：**

| 脚本 | 适用时机 | 检查什么 |
|------|----------|----------|
| `check_env.bat` | **启动前**（SD WebUI 没起 / 想确认配置对不对） | 路径、端口空闲、API 探活 |
| `diag.bat` | **运行中**（已经启动但有问题） | 端口占用、进程数、HTTP 健康 |

> 脚本是**只读**的，不会启动任何程序也不会修改文件，可以反复跑。

## 九、常见问题（FAQ）

**Q1：双击 run.bat 闪退或出现乱码 / `'xxx' 不是内部命令`？**
脚本已使用纯 ASCII 编写以规避 cmd.exe 的编码解析问题。若仍闪退，请改用命令行方式 B 运行以查看具体报错。

**Q2：启动报 `ModuleNotFoundError`？**
依赖未安装，执行 `pip install -r requirements.txt`。

**Q3：端口 5000 被占用（`Address already in use`）？**
先按"第六节"停止已有进程，或修改 `app.py` 末尾 `app.run(..., port=5000)` 中的端口号。

**Q4：调用生成接口报 `Unexpected UTF-8 BOM`？**
本版本已通过 `utf-8-sig` 读取修复该问题；如仍出现，请确认 `data/designs.json` 未损坏，可将其内容重置为 `[]`。

**Q5：SD WebUI 已启动但系统提示「SD WebUI 未连接，无法生成图片」？**
系统运行时动态检测 `GET /sdapi/v1/options`，每次请求都重新探活（无需重启 Flask）。
如果 WebUI 启动在 Flask 之后，刷新页面重试即可。若持续报错，运行 `diag.bat` 查看 API 状态码。

**Q6：输入「变形破损裂纹扭曲」等负面词后，4 张图都变成灰色 ⚠ 提示图？**
这是 **input_guard 工业负面词预检**生效的正常行为。系统会在调用文生图前先扫描 prompt，
命中 9 类工业敏感词（破 / 损 / 碎 / 脏 / 丑 / 坏 / 损伤 / 损坏 等，详见 `src/services/input_guard.py`）
时直接返回 SVG 拒绝图，不污染数据库。请改用「风格 / 颜色 / 杯型 / 场景」描述。

**Q7：4 张图要 30~60s，本地推理为什么这么慢？**
三种情况，按概率排序：
1. **首次启动**：PyTorch 正在编译 CUDA 算子（首次约 30~90s），第二次会快很多；
2. **`--xformers` 未开启**：在 `webui-user.bat` 的 `COMMANDLINE_ARGS` 末尾加 `--xformers` 可提速 20~40%；
3. **batch_size=4 一次出 4 张**：SD 一次出 4 张比 4 次出 1 张要快，但首张要等整 batch 完成。
可加 `--medvram`（4~6GB 显存）或 `--lowvram`（<4GB 显存）参数。

**Q8：Flask 控制台报 `SD WebUI 启用了页面但未开启 API`？**
这是 `test_connection()` 在 `/sdapi/v1/options` 收到 404 时给出的提示，说明 WebUI 跑起来了
但没启用 HTTP API。处理方式：编辑整合包里的 `webui-user.bat`，找到
`set COMMANDLINE_ARGS=` 这一行（可能为空），在末尾加上 `--api` 后保存并重启 WebUI。
秋葉整合包默认就带了 `--api`，一般不需要改。

**Q9：接入 SD WebUI 后 4 张图要 30~60s，本地推理为什么没比 Pollinations 快多少？**
三种情况，按概率排序：
1. **首次启动**：PyTorch 正在编译 CUDA 算子（首次约 30~90s），第二次会快很多；
2. **`--xformers` 未开启**：在 `webui-user.bat` 的 `COMMANDLINE_ARGS` 末尾加 `--xformers` 可提速 20~40%；
3. **batch_size=4 一次出 4 张**：SD 一次出 4 张比 4 次出 1 张要快，但首张要等整 batch 完成。
可加 `--medvram`（4~6GB 显存）或 `--lowvram`（<4GB 显存）参数。

**Q10：整合包路径和脚本里写的不一样？**
整合包路径变了怎么办？打开项目根目录的 `check_env.bat`，修改顶部 `SD_LAUNCHER` 变量为你整合包里
`A绘世启动器.exe` 的完整路径即可，其它逻辑无需调整。

| POST | `/api/img2img` | 图生图：multipart/form-data（起始图 + 强度 + prompt + sampler）|
| POST | `/api/optimize` | 对指定 design_id 执行 MOPSO 工艺优化 |
| GET | `/api/designs` | 获取全部设计方案 |
| GET | `/api/designs/<id>` | 获取单个设计方案 |
| PUT | `/api/designs/<id>` | 更新设计方案 |
| DELETE | `/api/designs/<id>` | 删除设计方案 |
| GET | `/api/designs/export/csv` | 导出全部设计方案为 CSV 文件 |
